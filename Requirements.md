# Правила работы с задачами

Статусы задач (Tasks):

1. Todo - можно брать задачу в работу.
3. In Progress - начата работа по задаче. Вы сами переносите задачу в столбец для этого статуса,
  когда начинаете выполнять эту задачу, или GitHub перенесёт её в этот столбец, когда вы создадите
  Pull Request.
4. Done - задача выполнена, проверена, ваш Pull Request включён в основную ветку (_master_). GitHub
  перенесёт задачу в этот столбец после вливания вашего Pull Request в ветку _master_.
5. Canceled - задача отменена, не будет выполнена. Только руководитель команды переносит задачу
  в этот статус.

## Информация для аналитика

Все функциональные и нефункциональные требования подразделяются на 4 типа записей:

* функция (Feature)
* история (User Story)
* сценарий (Use Case)
* задача (Task)

**Если нужно согласовывать документацию с англоязычным заказчиком:**  
Название каждой записи требований должно быть на английском языке. Описание может быть на русском
или английском.

Этапы подготовки документации по мере нарастания детализации требований:

1. Определить условия и функциональные требования с т.з. бизнеса, которые нужно выполнить (Feature).
  Пример: отправка уведомлений покупателю об изменении статуса заказа. Эти условия и требования
  обычно обозначены заказчиком и имеют ценность для продукта с т.з. заказчика.
2. Запись в формате User Story (роль пользователя, действие, цель действия) с привязкой к одной или
  нескольким записям типа Feature.
3. Добавление записей в формате Use Case и связывание их с записями Feature и User Story. Запись
  Use Case описывает реализацию одной или нескольких функций (Feature), а её участниками может быть
  несколько ролей пользователей. Значит, одна запись Use Case может быть связана с одной или
  несколькими записями типа User Story. Следует определять Use Case как отдельно выполняемый
  сценарий с поседовательными действиями одного или нескольких пользователей; причём действия
  складываются в сценарии для достижения определённой цели. Например, в сценарии "Оформление билета
  на авиарейс" продавец выставляет билеты на продажу, а покупатель -- выбирает и покупает билеты.
  Каждый Use Case должен заканчиваться измеримым результатом (другими словами: конечное состояние,
  постусловие).
4. Разбиение User Story и Use Case на составные части (задачи, Tasks). Каждая задача должна быть
  связана только с одним исполнителем и может быть завершена без ожидания выполнения других задач
  после того, как эта задача была начата (т.е. не должно быть параллельной работы в форме
  сопровождения одной задачи другой задачей). Одна задача может относиться к нескольким записям
  Use Case и User Story. Каждая задача получает описание необходимых действий с такой детализацией,
  которая нужна для выполнения Use Case и User Story и согласования с другими задачами, но не более
  этого. Напр., если нужно добавить во frontend-приложение отправку данных в backend-приложение, то
  контракт взаимодействия по API должен быть определён до передачи задачи frontend-разработчику.

Запись каждого типа (Feature, User Story, Use Case, Task) должна быть связана хотя бы с одной
записью другого типа. Только записи типа Task можно передавать на выполнение программистам. Если
запись Task связана с записями User Story или Use Case, то именно по этим записям надо тестировщикам
проверять решение Task.

Схема связей между типами задач:

```ditaa
        +------------+
   +--> | User Story | <-+
   |    +------------+   |
   v           ^         v
+---------+    |     +------+
| Feature |    |     | Task |
+---------+    |     +------+
   ^           v         ^
   |    +------------+   |
   +--> |  Use Case  | <-+
        +------------+
```

Все связи определены как "многие-ко-многим".

Во всех записях требований должны быть отражены условия готовности (или приёмки) реализации
требований, т.е. как можно убедиться в том (как протестировать), что решение соответствует
требованию.

Кроме этого, в записях типа Task должны быть:
1. Макеты пользовательского интерфейса, если нужно внести в него изменения.
2. Условия взаимодействия с другими системами, если нужно добавить или внести изменения в это взаимодействие. Напр., ссылка на схему OpenAPI и название используемой команды API.

## Информация для разработчиков

Если для завершения задачи нужно выполнить какой-то скрипт на сервере (напр., заполнение таблицы
в БД), то этот скрипт должен быть сохранён в общем хранилище команды разработки, а ссылка (или
команда для запуска) для него должна быть указана в комментарии к задаче.
