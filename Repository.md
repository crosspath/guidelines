# Правила работы с репозиторием

Ветки репозитория:

1. _master_ - основная ветка репозитория. Все остальные ветки создаём на основе _master_ и
  в Pull Request указываем, что хотим влить наши изменения в коде в ветку _master_.
2. _feature_-ветки:
  * если для идентификатора задачи используется формат БУКВЫ-ЦИФРЫ (как в JIRA, пример: JTI-166),
    тогда называем ветки так же, как идентификатор задачи, взятой в работу;
  * если есть только номер задачи, тогда называем ветки по шаблону "feature/{id}", например,
    "feature/123", где 123 - номер задачи, взятой в работу;
  * можно использовать идентификатор родительской задачи, если её подзадачи нельзя выложить до
    решения всех остальных подзадач - в этом случае решение подзадач "вливается" в Pull Request
    родительской задачи.

Не коммитим в ветку _master_! Все изменения в коде должны быть в _feature_-ветке и затем
в Pull Request.

Тексты коммитов должны быть на английском языке.

Pull Request должен быть связан с задачей на доске проекта, например, с автоматическим или ручным
добавлением ссылки из Pull Request на страницу задачи проекта.

Каждый Pull Request проходит этап Code Review. Все разработчики должны просматривать код друг друга
и либо добавлять комментарии о том, что можно или нужно сделать по-другому, либо о том, что согласны
с изменениями (достаточно одного слова - LGTM, либо использовать кнопку "Approve").

По возможности не создаём _feature_-ветки на основе других _feature_-веток. Если возникает такая
потребность, то стараемся убрать нежелательный код (не проверенный, не законченный) из
_feature_-ветки и затем влить её в _master_, а после этого создать ветку от _master_ для доработки
решения задачи.

По возможности используем опции "Squash commits" и "Delete branch after merge" в Pull Request.

Используем _merge_ как "Fast-forward merge", если это возможно, иначе как "Merge with commit".
По возможности не используем _rebase_.
