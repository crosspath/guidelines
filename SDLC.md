# Software development life cycle (жизненный цикл разработки программного обеспечения)

На основе модели <abbr title="Rational Unified Process">RUP</abbr> и опыта работы с заказчиками
напрямую.

Знаком `-->` показан изменённый или добавленный документ в ходе выполнения указанной стадии.

Стадии работы над задачей:

1. Выявление потребности. Определение времени, требуемого на исследование потребности и способов
  решения.

  --> Цели задачи
  --> Оценка времени на исследование

2. Уточнение требований у заказчика. Получение доступа к программному коду, если нужно доработать
  или интегрировать с ним решение задачи. Определение способов решения задачи. Варианты решений:
  административные и организационные меры, технические и программные решения. Определение возможных
  рисков и действий до и после наступления событий риска. Определение целевой аудитории,
  заинтересованных сторон и лиц, принимающих решения. Формирование и согласование плана по
  коммуникациям.

  --> Вопросы и предложения заказчику
  --> Записи обсуждений с заказчиком, протоколы встреч
  --> Доступ к существующему программному коду
  --> Запись типа Feature
  --> План действий по рискам
  --> Описание целевой аудитории
  --> Список заинтересованных сторон и лиц, принимающих решения
  --> План по коммуникациям

3. Подготовка коммерческого предложения с приблизительной оценкой срока и стоимости работ,
  составление и согласование договора.

  --> Коммерческое предложение
  --> Договор

4. Детализация технических и программных решений, создание прототипов решений. В качестве прототипа
  можно использовать макет пользовательского интерфейса.

  --> Прототипы решений
  --> Записи типа User Story
  --> Записи типа Use Case

5. Составление и согласование технического задания и сметы работ, выявление критериев приёмки и
  тестирования. Формирование требований для среды разработки на стороне заказчика (например,
  настройка сервера). Определение плана внедрения решения.

  --> Записи типа Task
  --> Смета работ
  --> Критерии приёмки и тестирования
  --> Требования для среды разработки на стороне заказчика
  --> План внедрения

6. Создание решения, например, в виде программного кода.

  --> Программный код

7. Проверка и доработка решения на данных, близких к реальным.

  --> Программный код

8. Приёмка и внедрение решения, передача решения в использование с реальными данными. Договор
  ограничивает количество итераций приёмки работ в случае, когда запросы на изменения не связаны
  с ошибками решения (например, ограничения действуют на изменение цвета и расположения элементов
  на экране).

  --> Инструкция по внедрению
  --> Пользовательская документация (например, инструкция по использованию решения)

9. Отслеживание работоспособности решения втечение согласованного срока гарантии работ.

  --> Отчёты по обнаруженным проблемам и принятым действиям

На этапах 6-8 происходит подготовка среды разработки на стороне заказчика:

* если нужно как можно раньше интегрировать решение задачи с существующим приложением, то
  подготовка выполняется на этапе 6;
* если передача результатов работы происходит только на стадии приёмки (этап 8), то подготовка
  выполняется на этапе 8.

Запросы на изменения после согласования технического задания и сметы работ:

1. Если изменение содержит только те требования, которые записаны в согласованных документах (т.е.
  нужна доработка в соответствии с согласованными условиями и требованиями задачи), то запрос на
  изменение принимается без дополнительной оплаты.
2. Дополнительные, ранее не учтённые требования проходят процесс оценки, начиная с этапа 1.
